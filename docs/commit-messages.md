# Правила коммитов и сообщений

## Формат заголовка

Базовый формат: `<type>(<scope>): <subject>`. Допустимый вариант без области: `<type>: <subject>`.

Примеры:

- `fix(components): исправить копирование ссылки в DetailsSummary`
- `docs(content): обновить правила для таблиц`
- `chore: обновить dev-зависимости`

## Обязательные правила

- `type` только в нижнем регистре;
- `type` не может быть пустым;
- `subject` не может быть пустым;
- длина заголовка `header` не больше 100 символов;
- перед `body` и `footer` нужна пустая строка.

## Допустимые `type`

- `feat` — новая функциональность;
- `fix` — исправление ошибки;
- `docs` — документация;
- `style` — изменения внешнего вида;
- `format` — форматирование кода без изменения логики и UI;
- `refactor` — рефакторинг без новой функциональности;
- `chore` — рутинные задачи и обслуживание проекта;
- `content` — изменения контента статей;
- `revert` — откат коммита;
- `merge` — слияние веток.

## Допустимые `scope`

- `app`
- `components`
- `styles`
- `hooks`
- `utils`
- `sections`
- `aefaq`
- `prfaq`
- `psfaq`
- `aeexpr`
- `rules`
- `links`
- `docs`
- `config`
- `deps`
- `assets`
- `*`

## Рекомендации по `subject`

- пишите кратко и предметно: что изменено;
- не дублируйте в `subject` `type` и `scope`;
- используйте глагол действия в нейтральной форме.

## Примеры корректных сообщений

```text
feat(app): добавить фильтр поиска по тегам
fix(hooks): исправить обработку hash в useAnchorValidation
docs(docs): обновить раздел про контентные компоненты
content(psfaq): добавить ответ про ошибку экспорта PNG
refactor(utils): упростить функцию нормализации ссылок
```

## Примеры некорректных сообщений

```text
Fix(components): ...
feature(app): ...
fix(unknown): ...
docs:
```

## Что проверяется автоматически

- в `pre-commit` запускается `lint-staged` (`.husky/pre-commit`);
- в `commit-msg` запускается `yarn commitlint --edit "${1}"` (`.husky/commit-msg`).
