# Правила коммитов и сообщений

## Формат заголовка

Базовый формат: `<type>(<scope>): <subject>`. Допустимый вариант без области: `<type>: <subject>`.

Примеры:

- `fix(components): исправлено копирование ссылки в спойлере`
- `docs(docs): обновлены правила для таблиц`
- `chore: обновлены зависимости для разработки`

## Обязательные правила

- `type` только в нижнем регистре;
- `type` не может быть пустым;
- `subject` не может быть пустым;
- длина заголовка `header` не больше 150 символов;
- перед `body` и `footer` нужна пустая строка;
- при перечислении нескольких `scope` используется формат `scope1, scope2`, то есть запятая и один пробел.

## Требования к языку и стилю сообщений

- Сообщения коммитов должны быть на русском языке;
- Сообщения должны начинаться с маленькой буквы и быть короткими;
- `subject` пишется в форме результата без точки в конце, например, `добавлен`, `исправлено`, `обновлены`, `удалены`, `вынесены`;
- Запрещено использовать прямые названия файлов и переменных в сообщении коммита.

## Атомарность коммитов и разделение изменений

- Один логический блок изменений должен соответствовать одному атомарному коммиту;
- Нельзя объединять все правки и файлы в один коммит разом, если они логически разделяются. Их необходимо разделять строго по смыслу;
- При необходимости деления одного файла на несколько независимых коммитов используются временные файлы с расширением `.bak`. Временные `.bak` файлы используются только для безопасного разбиения, они не должны добавляться в коммит и должны быть удалены после завершения разбиения.

## Допустимые `type`

- `feat` — новая функциональность;
- `fix` — исправление ошибки;
- `docs` — изменения в документации;
- `format` — правки форматирования кода, не влияющие на логику или внешний вид;
- `refactor` — рефакторинг кода без исправления ошибок или добавления новой функциональности;
- `chore` — рутинные задачи, обновление зависимостей, настройка, сборка;
- `content` — изменения в контенте статей;
- `revert` — откат предыдущего коммита;
- `merge` — слияние ветвей.

## Допустимые `scope`

- `app` — общая часть приложения;
- `components` — компоненты интерфейса;
- `styles` — стили и SCSS слои;
- `hooks` — прикладные хуки;
- `utils` — утилиты;
- `aefaq` — контент раздела AE FAQ;
- `prfaq` — контент раздела PR FAQ;
- `psfaq` — контент раздела PS FAQ;
- `aeexpr` — контент раздела AE Expressions;
- `rules` — правила чата и подобные страницы;
- `reg` — регулярные выражения и файлы реестра;
- `links` — ссылки;
- `docs` — документация проекта;
- `config` — конфигурационные файлы проекта;
- `deps` — зависимости проекта;
- `assets` — медиафайлы и ресурсы;
- `scripts` — служебные скрипты;
- `*` — глобальные изменения или отсутствие конкретной области.

## Исправление и отправка коммитов

- Запрещено использовать параметр `--no-verify`;
- Для исправления точечных ошибок в уже сделанных, но ещё не отправленных коммитах, следует использовать команду `git commit --fixup <commit_hash>`, чтобы затем объединить их через rebase с автосквошем.

## Автоматическая проверка

- В `pre-commit` запускается `lint-staged` из `.husky/pre-commit`;
- В `commit-msg` запускается проверка заголовка коммита через `yarn commitlint --edit "${1}"` из `.husky/commit-msg`.
