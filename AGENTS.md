# AGENTS.md

## Purpose

Этот файл задает рабочие правила для AI-агента в репозитории `aechat-links`. Цель: вносить минимальные и безопасные изменения под задачу пользователя. Язык коммуникации по умолчанию: русский.

## Workflow

1. Перед любыми правками сначала проанализировать дерево проекта и кратко зафиксировать, какие директории/файлы затрагиваются.
2. Перед изменениями в коде, контенте или процессах проверить релевантную документацию в `docs`.
3. Не начинать правки до анализа структуры и связей целевых файлов.
4. Если изменение большое, заранее предложить план разбиения на 2-4 атомарных коммита.

## Command Policy

1. Не запускать без явного запроса пользователя: `build`, `dev`, `test`, `lint`, `format`, `install` и другие ресурсоемкие команды.
2. Не выполнять сетевые команды и установку пакетов без явного запроса пользователя.
3. Сборку проекта не запускать без явного запроса.
4. Если для задачи нужна индексация проекта, выполнять только при необходимости и явно указывать это в отчете.
5. При недостатке локального контекста и необходимости решения проблемы разрешено обращаться к интернет-источникам.
6. Форматирование выполнять точечно по файлу через yarn-команды: `yarn prettier --write <file>`, `yarn eslint --fix <file>`, `yarn stylelint --fix <file>`.
7. Перед форматированием целевого файла по возможности удалять лишние пустые строки в редактируемом файле.
8. Для удаления пустых строк использовать `node scripts/removeEmptyLines.js <путь_к_файлу>` (поддерживается точечный запуск по файлу или директории).
9. `yarn format` сейчас запускает `tsc + stylelint + eslint + prettier` по всему проекту; не использовать его без явного запроса пользователя.

## Project Layout (Important)

1. Статьи находятся в `src/pages/sections`.
2. Основные разделы статей: `src/pages/sections/aefaq`, `src/pages/sections/prfaq`, `src/pages/sections/psfaq`, `src/pages/sections/aeexpr`.
3. Не изменять структуру разделов и имена файлов в `src/pages/sections/*` без явной необходимости.

## Code Change Rules

1. Не перемещать код между файлами и модулями без явной необходимости задачи.
2. Не делать массовые косметические изменения и рефакторинг, не относящиеся к задаче.
3. Не трогать артефакты и внешние каталоги (`dist`, `node_modules`), если это не требуется задачей.
4. Не изменять существующие незакоммиченные изменения в рабочем дереве без острой необходимости задачи и явного обоснования.
5. Одно логическое изменение = один коммит.
6. Делать изменения минималистично, если это позволяет задача; не раздувать кодовую базу без необходимости.
7. При правках учитывать стиль и паттерны ближайших похожих файлов в проекте.
8. Не дублировать код между файлами и модулями; при повторении логики выносить общий код в утилиты/общие функции (дедупликация) при сохранении текущего поведения.
9. Если новая логика потенциально используется в нескольких местах, по умолчанию проектировать её как переиспользуемую, а не копировать реализацию.
10. Не использовать `index.tsx`/`index.ts` как файл-реэкспорт или точку входа; использовать явные имена файлов модулей.
11. Запрещено допускать спагетти-код: не смешивать в одном модуле UI, бизнес-логику, ранжирование/алгоритмы, побочные эффекты и низкоуровневые утилиты без четкого разделения ответственности.
12. Если файл становится трудночитаемым или избыточно связанным, обязательно выносить независимые части в отдельные модули (например: `components`, `hooks`, `utilities`, `config`) без изменения функционала.

## Naming Rules

1. Имена файлов и директорий писать в `camelCase` или `PascalCase` по существующим паттернам проекта; для React-компонентов использовать `PascalCase` (`SearchModal.tsx`), для утилит и хелперов `camelCase` (`searchScoringUtilities.ts`).
2. Не использовать сокращения и неочевидные аббревиатуры в именах файлов, переменных и функций, если это не общепринятый термин.
3. Переменные и функции писать в `camelCase`; булевые переменные начинать с `is/has/can/should`.
4. Типы, интерфейсы, enum и React-компоненты писать в `PascalCase`.
5. Константы верхнего уровня (в том числе веса/пороги/regex-константы) писать в `UPPER_SNAKE_CASE`.
6. Имена generic-типов делать осмысленными (`TItem`, `TResult`), не использовать одиночные буквы без необходимости.
7. CSS/SCSS-классы именовать в едином стиле текущего файла/модуля; новые классы не создавать в конфликтующем стиле.
8. Названия обработчиков событий в компонентах начинать с `handle` (`handleClick`, `handleQueryChange`), коллбеки-пропсы начинать с `on` (`onClose`, `onSelect`).
9. Не использовать транслит в идентификаторах, именах файлов и именах CSS-классов; только английские термины.
10. Перед созданием нового имени проверять ближайшие файлы на уже принятые паттерны и повторно использовать существующие naming-конвенции.

## Documentation Rules

1. Перед изменениями использовать `docs` как источник проектных правил.
2. Если изменения меняют поведение, структуру или договоренности проекта, обновить соответствующую документацию в `docs`.
3. Если обновлены документация или конфигурационные файлы проекта, проверить актуальность `AGENTS.md` и обновить его при необходимости.
4. Разрешено синхронизировать `commitlint.config.js` с текущим деревом проекта и фактическими зонами изменений.
5. При синхронизации `commitlint.config.js` не перегружать конфиг: поддерживать короткий и практичный список `type/scope`, без избыточной детализации.
6. В русскоязычных заголовках и обычном тексте использовать нормы русского языка: с заглавной буквы писать только первое слово предложения/заголовка и собственные имена; остальные слова писать со строчной буквы.

## Decision And Safety Rules

1. Приоритет инструкций: прямой запрос пользователя > `AGENTS.md` > локальные комментарии в файлах.
2. Перед правками показывать текущее состояние `git status --short` и список целевых файлов.
3. Если изменение рискованное или неоднозначное, сначала предложить 2-3 варианта решения с краткими trade-off.
4. Для каждой значимой правки фиксировать не только что изменено, но и зачем это сделано.
5. Не менять публичные API, контракты, сигнатуры и поведение внешних интерфейсов без явного согласования.
6. При технических задачах не менять смысл контента статей без отдельного запроса.
7. При изменениях, связанных с поиском или индексацией, проверять, что якоря и внутренние ссылки не сломаны.
8. Не изменять файлы вне области задачи, даже если найдены несвязанные проблемы.
9. При критичной неопределенности сначала задать один уточняющий вопрос, а не делать рискованное предположение.
10. В итоговом отчете явно указывать ограничения проверки и что именно не было проверено.
11. Если документация в `docs` расходится с фактическим кодом, сначала сообщить о расхождении и только потом предлагать изменение.
12. Для больших задач сначала давать краткий пошаговый план и только потом переходить к правкам.
13. Не переименовывать файлы и директории без отдельного запроса.
14. Если переименование согласовано и действительно нужно, выполнять его через `git mv`, а не через `rename`/ручное перемещение.
15. Не менять lock-файлы и конфигурационные файлы, если задача не связана с зависимостями или конфигом.
16. В отчете по изменениям указывать возможные побочные эффекты и зоны риска.
17. Разрешено менять `AGENTS.md` по явному запросу пользователя; также разрешено proactively предлагать изменения в `AGENTS.md`, если текущие правила мешают работе и пользователь явно раздражен.
18. Если пользователь добавляет дополнительный контекст или новую подзадачу, не прерывать текущую задачу, пока пользователь явно не попросит переключиться.

## Execution Quality Rules

1. Definition of done: задача завершена только после внесения нужных правок, актуализации релевантной `docs` и итогового отчета с рисками.
2. После согласования плана не расширять scope задачи без отдельного подтверждения пользователя.
3. Не выполнять скрытые попутные рефакторы; несвязанные улучшения выносить в отдельную задачу.
4. Перед правками явно фиксировать ключевые допущения, если они есть.
5. При изменении поведения явно указывать, есть ли риск нарушения обратной совместимости.
6. В отчетах и обсуждении использовать пути относительно корня репозитория (`src/...`, `docs/...`).
7. Не изменять `.env*`, секреты, ключи, токены и настройки CI с чувствительными данными без явного запроса.
8. Не редактировать вручную автогенерируемые файлы, если это не отдельная согласованная задача.
9. Для рискованных изменений указывать краткий вариант отката.
10. Даже если тесты/сборка не запускаются, выполнять минимум статической проверки измененных файлов (импорты, типы, ссылки, якоря).
11. Перед значимыми правками кратко фиксировать expected diff в формате «было/станет» по затрагиваемым файлам.
12. Если обнаружены неожиданные чужие изменения в затрагиваемых файлах, остановиться и запросить подтверждение перед продолжением.
13. Если пользователь меняет файлы извне во время работы агента, агент обязан повторно синхронизировать состояние через `git status --short`; при пересечении с текущей задачей остановиться и запросить подтверждение.
14. Любой создаваемый и изменяемый код должен соответствовать актуальным конфигурациям проекта: `eslint.config.js`, `.stylelintrc.json`, `.prettierrc.json`, `tsconfig*.json`, `commitlint.config.js` и другим релевантным конфигам в корне репозитория.
15. Перед завершением задачи выполнять минимум статической проверки на соответствие этим конфигам для затронутых файлов и явно указывать ограничения, если полная проверка не запускалась.

## Git And Commit Rules

1. Коммиты только атомарные (1 логическое изменение = 1 commit).
2. Commit message строго по `commitlint.config.js`.
3. Перед выполнением commit и перед разбиением изменений на несколько commit сначала предлагать пользователю план коммитов и порядок их выполнения.
4. Формат commit message: `<type>(<scope>): <русский текст в форме результата>`.
5. Текст commit message на русском, в форме результата: `добавлен/исправлено/оптимизировано/убран`.
6. В commit message по возможности использовать смысловые формулировки вместо сырых имен файлов/скриптов.
7. Никогда не использовать `--no-verify`.
8. Перед каждым commit показать `git diff --staged --name-only` и кратко что именно коммитится.
9. Если pre-commit падает, исправить замечания и повторить commit без `--no-verify`.
10. Коммиты должны соответствовать стилю минимум 15 последних коммитов проекта.
11. Соблюдать доступные `type/scope` из `commitlint.config.js`.
12. Не объединять несвязанные правки в один коммит.
13. Если в рамках одного запроса выполнено несколько независимых задач, делать отдельный коммит на каждую задачу.
14. Учитывать хуки репозитория: `pre-commit` запускает `npx lint-staged`, `commit-msg` запускает `yarn commitlint --edit`.
15. Не выполнять `git add`, `git commit` и другие шаги фиксации изменений, пока пользователь явно не подтвердит запуск коммитов отдельным сообщением после предложенного плана.
16. План коммитов предлагать в коротком шаблоне со служебными словами:

- `Файл: <путь>`
- `---`
- `Коммит: <один вариант commit message>`
- `Изменения: <кратко, что изменено в рамках задачи>`

17. Если один файл содержит несколько независимых логических правок, по возможности разбивать их на отдельные атомарные коммиты.
18. Для аккуратного разбиения изменений внутри одного файла разрешено временно использовать дубликат файла в формате `.bak`, если это действительно помогает безопасно и прозрачно разделить коммиты.
19. Перед коммитом изменений в одном файле отдельно анализировать, можно ли разделить правки на несколько независимых смысловых блоков (например: лимиты, scope, кастомные правила) и предлагать такое разбиение в плане коммитов.
20. Глагол в заголовке commit message должен строго соответствовать фактическому действию: `добавлен/вынесены/перенесены/обновлены/исправлены/удалены`; не использовать обобщающее `оптимизировано`, если в коммите нет реальной оптимизации производительности или алгоритма.

## Unified Commit Style

1. Базовый формат: `<type>(<scope>): <результат на русском языке>`.
2. Допустимая длина заголовка: до 150 символов (по `commitlint.config.js`).
3. Предпочтительные `type` по фактической частоте проекта: `content`, `refactor`, `fix`, `feat`, затем `chore`, `style`, `format`.
4. Предпочтительные `scope` по фактической частоте проекта: `components`, `aefaq`, `styles`, `hooks`, `prfaq`, `docs`, `app`, `psfaq`, `config`, `deps`, `utils`, `scripts`.
5. Subject писать в форме результата (что уже сделано), без точки в конце и без англоязычного imperative.
6. Для технических задач по умолчанию использовать `fix/refactor/feat` с конкретным scope; `chore(config|deps)` только для инфраструктуры.
7. Для статей и текстов по умолчанию использовать `content(<aefaq|prfaq|psfaq|aeexpr>)`; `fix(<aefaq|prfaq|psfaq|aeexpr>)` только если это именно исправление ошибки.
8. Не использовать `[WIP]` в commit message; незавершенную работу не коммитить в основную историю.
9. `merge:` и `revert:` использовать только когда это реально merge/revert-операция, а не обычное изменение.
10. Если scope неочевиден, выбрать самый узкий из `commitlint.config.js`, а не общий `*`.

## Reporting Format

После завершения задачи кратко указать:

1. Какие файлы изменены.
2. Что именно сделано.
3. Почему изменения минимальны и достаточны для задачи.
4. Выполнялась ли индексация проекта и зачем.
