# AGENTS.md

## Назначение

Правила работы AI-агента в репозитории `aechat-links`. Цель: вносить минимальные безопасные изменения под задачу пользователя. Язык по умолчанию: русский.

## Короткая коммуникация

1. Писать кратко и по делу: решение, измененные файлы, проверки, ограничения.
2. Не пересказывать очевидное и не дублировать один и тот же риск в нескольких местах.
3. Промежуточные сообщения нужны только для статуса, важных находок, рисков или перед правками.
4. Если задача простая, итоговый ответ должен быть коротким списком без длинного вступления.
5. Если задача большая, сначала дать компактный план, затем выполнять только согласованный scope.

## Рабочий порядок и безопасность

1. Приоритет: прямой запрос пользователя > `AGENTS.md` > локальные комментарии в файлах.
2. До правок изучить структуру и связи целевых файлов, а также проверить релевантную документацию в `docs`. Учитывать контекст текущих изменённых файлов при планировании и выполнении правок.
3. Перед значимыми правками показать `git status --short` и список целевых файлов.
4. До выполнения правок кратко согласовать подход или 2-3 варианта с trade-off, если решение неоднозначное. Не расширять scope после согласования без отдельного подтверждения.
5. Если найдены неожиданные чужие изменения в целевых файлах (или пользователь меняет файлы во время работы), остановиться и спросить пользователя (повторно проверить `git status --short`).
6. Во время работы вести историю действий и правок для разложения изменений на атомарные коммиты (не превращая её в длинный отчет).
7. Предпочтительнее использовать точечный поиск по файлу (`grep_search`) и частичное чтение кода (через `view_file` с указанием конкретных `StartLine`/`EndLine` до 150 строк), чем читать большие файлы целиком, для экономии контекста модели.
8. Рискованные изменения сопровождать кратким вариантом отката.
9. Перед завершением выполнить минимум статической проверки измененных файлов: импорты, типы, ссылки, якоря. Если полная проверка не запускалась, явно указать это в итоговом отчете.

## Команды

1. Без явного запроса не запускать ресурсоемкие команды: `build`, `dev`, `test`, `lint`, `format`, `install`.
2. Без явного запроса не выполнять сетевые команды и установку пакетов.
3. `yarn format` запускает `tsc + stylelint + eslint + prettier` по всему проекту; не использовать без явного запроса.
4. Перед форматированием по возможности убрать лишние пустые строки через `node scripts/maintenance/removeEmptyLines.js <file>`.

## Структура проекта

1. Статьи находятся в `src/pages/sections`.
2. Основные разделы статей: `aefaq`, `prfaq`, `psfaq`, `aeexpr`.
3. Основные каталоги кода:
   - `src/components/content` - компоненты статей;
   - `src/components/features` - интерактивные инструменты;
   - `src/components/layout` - шапка, футер, переходы;
   - `src/hooks` - прикладные хуки;
   - `src/utils` - утилиты;
   - `src/styles` - SCSS-слои.
4. Не менять структуру `src/pages/sections/*` и имена файлов без явной необходимости.

## Правки кода

1. Делать минимальные изменения, достаточные для задачи.
2. Не делать попутный рефакторинг, массовую косметику и перемещение кода без необходимости.
3. Не трогать `dist`, `node_modules`, автогенерируемые файлы, `.env*`, секреты и lock-файлы без прямого запроса.
4. Не менять публичные API, контракты, сигнатуры и внешнее поведение без согласования.
5. Не менять смысл статей при технических задачах.
6. Учитывать стиль ближайших похожих файлов.
7. Не дублировать код; повторяемую логику выносить в общие функции при сохранении поведения.
8. Не использовать `index.ts`/`index.tsx` как реэкспорт или точку входа.
9. Разделять UI, бизнес-логику, алгоритмы, эффекты и низкоуровневые утилиты.
10. Если файл становится трудночитаемым, выносить независимые части в `components`, `hooks`, `utilities` или `config`, если это нужно задаче.
11. Если нужно переместить файл, использовать строго `git mv`. Если `git mv` не получается или не подходит для конкретного случая, сначала попросить пользователя подтвердить ручной перенос.
12. Строго следовать правилам линтеров (ESLint, SonarJS, Stylelint). Запрещено обходить их или пытаться отключать правила локально (`eslint-disable`) или глобально. Все предупреждения линтеров должны исправляться структурно через качественный код (например, мемоизацией именованных компонентов, переприсваиванием булевых параметров в константы).
13. При добавлении таймеров, дебаунсов, слушателей событий (например, resize/scroll) или подписок в React-компонентах, всегда реализовывать их очистку/сброс в return-функции `useEffect`.
14. В SCSS-модулях (`*.module.scss`) избегать переопределений через `:global` без крайней необходимости. Все стили должны оставаться локальными.
15. В логических функциях отдавать предпочтение паттерну Early Exit (ранний возврат / guard clauses) вместо глубокой вложенности блоков `if-else`.

## Именование

1. React-компоненты и `.tsx` файлы: `PascalCase`.
2. Утилиты, хуки и `.ts` файлы: `camelCase`.
3. Переменные и функции: `camelCase`; булевые - с `is/has/can/should`.
4. Типы, интерфейсы, enum и компоненты: `PascalCase`.
5. Константы верхнего уровня: `UPPER_SNAKE_CASE`.
6. Props-интерфейсы использовать с суффиксом `Properties`.
7. Не использовать транслит и неочевидные сокращения.
8. CSS/SCSS-классы называть в стиле текущего файла.

## Документация

1. `docs` - источник проектных правил перед правками.
2. При внесении правок обязательно обновлять или добавлять релевантную документацию в docs, если изменения затрагивают поведение, структуру или договоренности проекта.
3. Если изменены docs или конфиги, проверить, не нужно ли обновить `AGENTS.md`.
4. В русскоязычных заголовках писать с заглавной буквы только первое слово и собственные имена.
5. Если `docs` расходятся с кодом, сначала сообщить о расхождении, затем предложить или внести точечное исправление.
6. Вся техническая документация в каталоге `docs/` оформляется строго в едином стиле сплошного связного повествования на русском языке.
7. Запрещено использовать любые маркированные или нумерованные списки в текстовых описаниях (за исключением блоков кода).
8. Текст должен разделяться на короткие логические абзацы по 2-4 предложения, отделенные двойным переносом строки (пустой строкой) для удобочитаемости.
9. При упоминании других файлов документации или исходных файлов проекта обязательно проставляются перекрестные ссылки.

## Контент и ссылки

1. Для `src/pages/sections/**` корневой контейнер статьи: `<div className="article-content">`.
2. Один вопрос - один `DetailsSummary` с коротким `title` и стабильным `anchor` в `kebab-case`.
3. Для повторяющегося ответа предпочтительна ссылка на существующий `anchor`, а не копипаст.
4. При изменениях поиска, якорей или секций проверять внутренние ссылки и hash-навигацию.
5. Перед новым паттерном контента проверить готовые компоненты: `Addition`, `ArticleMedia`, `CodeSnippet`, `ContentFilter`, `NestedDetailsSummary`, `HostsAdobe`.
6. Текстовые ссылки в статьях и на страницах не должны быть длинными, кроме ссылок в виде вопросов. Для конструкций вроде «по этой ссылке» в теге `<a>` оставляется только слово «этой», а остальные слова выносятся наружу.

## Коммиты

1. Один логический блок изменений = один атомарный коммит.
2. Не выполнять `git add`, `git commit`, rebase и другие шаги фиксации без отдельного явного подтверждения.
3. Предлагать план коммитов и порядок только при явном запросе пользователя. Не засорять вывод предложениями о создании коммитов по умолчанию.
4. Не пихать все правки и файлы в один коммит разом, если они логически разделяются. Разделять их строго по смыслу.
5. При необходимости делать временные бэкапы через `.bak` файлы, чтобы раскладывать изменения поштучно. Если один файл содержит несколько независимых правок, обязательно использовать `.bak` перед его разбиением.
6. Временные `.bak`/backup-файлы использовать только для безопасного разбиения, не добавлять их в коммит и удалить после завершения разбиения.
7. При плане коммитов опираться на историю действий: путь файла, логический блок, цель правки, рекомендуемый commit message.
8. Commit message: `<type>(<scope>): <результат на русском языке>`. Не использовать прямые названия файлов и переменных в сообщении коммита.
9. Subject в форме результата без точки: `добавлен`, `исправлено`, `обновлены`, `удалены`, `вынесены`.
10. Не использовать `--no-verify`.
11. Перед commit показать `git diff --staged --name-only` и кратко что коммитится.
12. Если hook падает, исправить замечания и повторить commit без обхода hooks.
13. Ориентироваться на `commitlint.config.js` и последние коммиты.
14. Предпочтительные type: `content`, `refactor`, `fix`, `feat`, затем `docs`, `chore`, `style`, `format`.
15. Предпочтительные scope: `components`, `aefaq`, `styles`, `hooks`, `prfaq`, `docs`, `app`, `psfaq`, `config`, `deps`, `utils`, `scripts`.
16. Для исправления точечных ошибок в уже сделанных, но ещё не отправленных коммитах, использовать `git commit --fixup <commit_hash>`, чтобы затем объединить их через rebase с автосквошем.

Короткий шаблон плана коммита:

- `Файл: <путь>`
- `---`
- `Коммит: <type>(<scope>): <сообщение>`
- `Изменения: <кратко>`

1. Какие файлы изменены.
2. Что сделано и почему этого достаточно.
3. Какие проверки выполнены.
4. Что не проверялось и какие есть риски.
5. Выполнялась ли индексация проекта и зачем.
